# MolOpt Benchmark Summary

## 1. Oracle Definitions

The benchmark oracles are MolOpt-compatible wrappers around LIDDIA evaluator
tools.

Raw values come from `evaluator.tools` where available:

- `calculate_qed`
- `calculate_logp`
- `calculate_molecular_weight`
- `calculate_tpsa`
- `calculate_hbd`
- `calculate_hba`
- `calculate_rot_bonds`
- `predict_toxicity`

MolOpt expects a scalar score where higher is better. Some evaluator outputs
therefore need a direction transform:

| Oracle | Evaluator meaning | Benchmark score meaning |
| --- | --- | --- |
| `qed` | QED, higher is better | unchanged |
| `sascore` | raw SA score, lower is easier | `10 - SA`, higher is easier synthesis |
| `herg`, `dili`, `clintox`, `mutagenicity`, `carcinogens` | predicted toxicity probability, lower is safer | `1 - toxicity`, higher is safer |
| `logp` | raw LogP | clipped at zero for maximize-style benchmark plots |
| `mol_wt`, `tpsa`, `hbd`, `hba`, `rot_bonds` | raw physicochemical values | unchanged raw values |

SAScore note: in this environment, `evaluator.tools.calculate_sascore` returns
`None` because the RDKit SA scorer is not importable. The wrapper first tries
the evaluator call, then falls back to the MolOpt `fpscores.pkl` implementation.

We also ran an evaluator-backed rescoring pass. It preserved the original
generated molecules and oracle-call numbers, but recomputed scores with the
evaluator-backed wrappers. The canonical plots in `plots/full` and
`plots/medium` now use these evaluator-backed scores. The full 10K
QED/LogP/SAScore plots did not change. In the medium matrix, only molecular
weight shifted meaningfully because the evaluator uses exact molecular weight
while the first wrapper used average molecular weight.

## 2. Why We Benchmarked

The benchmark was run to inform lead-optimizer agent behavior.

The agent needs to know:

- which MolOpt algorithms are reasonable defaults for different oracle classes,
- how expensive each class of oracle is,
- whether optimizers can follow a supplied LIDDIA scoring function,
- whether an oracle is safe to maximize directly,
- which results should be treated as strong evidence versus a small scout.

The benchmark is tool-routing evidence. It is not a final molecular discovery
campaign.

## 3. Test Dataset

All runs used the same starter set:

```text
inputs/zinc_sanity_1k.smi
```

Dataset details:

- 1,000 starting SMILES
- sampled from the MolOpt/ZINC starter data
- prepared with `prepare_zinc_starters.py`
- sampling seed: `0`

Seeds in the benchmark are stochastic replicate runs over the same starter
file. They are not different slices of the 1,000 molecules.

This design tests optimizer/oracle behavior under matched starting conditions.
It does not prove broad chemical-space coverage.

## 4. Runs, Calls, Time, And Resources

### Completed Runs

| Run set | Oracles | Algorithms | Seeds | Oracle calls |
| --- | --- | --- | ---: | ---: |
| Medium cheap matrix | 8 cheap oracles | 11 algorithms | 3 | 1,000 |
| Full cheap pilot | QED, LogP, SAScore | 5 algorithms | 5 | 10,000 |
| Small ADMET scout | 5 tox endpoints | 11 algorithms | 1 | 100 |

The main plotted metric is top-k average over oracle calls:

- `top1`: best molecule found so far
- `top10`: average score of the best 10 molecules found so far
- `top100`: average score of the best 100 molecules found so far

For agent integration, `top10` is the most useful default because it reflects
shortlist quality rather than a single lucky hit.

### Slurm Resource Shape

Most jobs used:

- 1 node
- 4 CPU cores
- 1 A100 GPU requested
- 16 GB requested memory
- 12-hour walltime

| Full 10K SAScore pilot | 3:57:18 | 25 seed runs |

Cheap-oracle runs are feasible for broad sweeps. ADMET/toxicity runs are much

### Calls To Useful Values

Examples from the full 10K pilot, using mean top10 curves:

| Objective | Target | Fastest methods |
| --- | ---: | --- |
| QED | 0.94 | STONED: 567 calls; SMILES GA: 823; GPBO: 2818 |
| QED | 0.945 | STONED: 962 calls; GPBO: 5256 |
| LogP | 20 | SMILES GA: 3388 calls; STONED: 5887; GPBO: 8604 |
| SAScore desirability | 8.9 | Graph GA: 532 calls; GPBO: 2734 |

5.82 hours, and SAScore about 3.96 hours across the completed 10K pilot runs.

## 5. High-Level Performance Takeaways

### By Property Class

- QED is bounded and saturates. Many methods reach similar final values, so
  early efficiency and shortlist quality matter more than final top1.
- LogP is exploitable. String/generative methods can drive it very high, but
  this does not automatically mean better lead-like chemistry.
- SAScore desirability is easy for some methods to saturate in this transformed
  benchmark. Graph GA and GPBO reached the score ceiling on top10.
- Raw molecular weight, TPSA, HBA/HBD, and rotatable bonds are not safe
  standalone maximization goals for lead optimization. Optimizers can follow
  them, but the result may be chemically undesirable.
- ADMET/toxicity scores are usable as safety desirability scores after the
  `1 - toxicity` transform, but they are expensive and should be run
  selectively.

### By Algorithm

| Algorithm | Practical read from these runs |
| --- | --- |
| Screening | Good baseline. Strong when the starter set already contains high-scoring molecules. |
| Graph GA | Reliable general-purpose graph optimizer; strong on SAScore desirability. |
| SMILES GA | Strong early efficiency on exploitable cheap objectives such as LogP. |
| STONED | Strong full-budget performance on QED and LogP. |
| GPBO | Strong on cheap scalar objectives; useful but can have overhead. |
| Graph MCTS | Mixed on cheap objectives; strong in some small ADMET scout endpoints. |
| MolDQN | Mixed; can run, but not a default first choice from these results. |
| REINVENT / REINVENT_SELFIES | Useful but slower; route selectively. |
| MIMOSA | Mixed; showed useful behavior in some ADMET scout endpoints. |
| SelfiesGA | Mixed; useful for comparison, not yet a universal default. |

These are routing notes, not final rankings.

## 6. Limitations For Agent Integration

- The cheap-oracle matrix is stronger evidence than the small ADMET scout
  because it has 3 seeds. The small ADMET scout has only 1 seed.
- A high raw property score can mean oracle exploitation, not better lead
  optimization.
- ADMET/toxicity benchmarks are expensive because the evaluator model is called
  repeatedly for novel molecules.
- The current MolOpt interface scores molecules one at a time through the oracle
  callable. Duplicate SMILES can be cached by MolOpt, but unique SMILES still
  incur new evaluator calls.
- Full production lead optimization should use constraints, target ranges, or
  composite objectives rather than direct maximization of every descriptor.
- These runs use a 1,000-SMILES starter file. They are good for controlled tool
  comparison, not for claiming complete coverage of chemical space.

## 7. How A MolOpt Benchmark Run Works

1. A YAML config selects the tier, oracles, algorithms, starter file, output
   root, and skip/error behavior.
2. `run_molopt_from_config.sh` starts a Slurm job and activates the
   `molopt-liddia` conda environment.
3. `run_molopt_benchmark_config.py` reads the YAML config and builds a command
   for `run_molopt_oracle_tests.py`.
4. `run_molopt_oracle_tests.py` imports oracle callables from
   `liddia_oracles.py`.
5. `liddia_oracles.py` imports evaluator modules from `evaluator.tools`.
   Physicochemical oracles call evaluator functions directly. Toxicity oracles
   call `predict_toxicity` and then convert toxicity probability to safety
   desirability with `1 - toxicity`.
6. For each oracle, algorithm, and seed, MolOpt generates or selects molecules
   and calls the oracle function.
7. Results are saved as YAML files under:

```text
oracle_benchmark_results/<tier>/<oracle>/<algorithm>/seed_<n>/
```

8. Plotting and summary scripts read those YAML files and compute top1, top10,
   top100, AUC, and call-to-threshold tables.

For ADMET/toxicity, evaluator model loading happens inside the evaluator tool
module. The model is reused inside a Python process, but scoring many unique
molecules is still slow because predictions are made repeatedly through the
model ensemble.
